MVC - Model View Controller
  • !! The model is responsible for all the logic.
  • Controller is responsible for user input, e.g. checking for syntax error;
    Model is responsible for the logic, also checking logic errors in the input;
    !! But it is controller who makes sure the user is displayed error message;
  • The model has to communicate with view: it can be done either by:
    - sending data encapsulated in an object - but then please do not include any logic in the object, it is just for data sending (as the view will have the access to all object's methods);
    - by defining some other protocol defined, like sending just pure data (better);
  • The Model does not send the data to View - it puts the data 'somewhere', and notifies all views whenever its data has been changed; the views know where to get the data from; this is because we may have many views and just one model, it is not a good idea to send the data to all views each time;
In the undergraduate course they're learning the design patterns, and how to write down an application's architecture, how to do reverse engineering, how to implement a pattern or recognize what is the architecture type; but in the graduate course you will also learn how do you know whether an architecture is good or bad, or which one is better;

In Taks 4, when asked if now you would come up with different architecture for the your certain old application and why, you should think of if you're able to improve the quality by changing the architecture; by improving quality I mean improving one of the 6 quality components, which were modifiability, security, etc.
